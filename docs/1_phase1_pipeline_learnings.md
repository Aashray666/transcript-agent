# RiskMapper Pipeline — Learnings & Limitations

## V1 vs V2 Comparison

| Metric | V1 Pipeline | V2 Pipeline |
|---|---|---|
| Raw mentions extracted | 68 (10/12 chunks) | 84 (15/15 questions) |
| After dedup | 18 risks | 58 risks |
| Dedup effectiveness | 74% reduction | 31% reduction |
| Unmapped | 1 (rate limit) | 2 |
| Total API calls | ~25 | ~75 |
| Total time | ~4.5 min | ~7.5 min |
| Quality (manual review) | 10/18 solid, 8 issues | First 15 solid, rest are cascade noise |

**V1 won on output quality.** Fewer chunks meant fewer duplicate extractions, and the dedup had fewer items to reason about — so it merged more aggressively and correctly.

**V2 won on extraction completeness.** Per-question parsing caught risks V1 missed (data privacy from Q3, spectrum from Q3, data incident from Q5, infrastructure sharing from Q6). But the dedup couldn't collapse 84 items effectively.

## LLM Calls — Where, Which Model, and Why

### Step 1: Registry Loading
- **LLM calls:** 0
- **Why:** Pure data loading. XLSX → ChromaDB. Embeddings generated by ChromaDB's default Sentence Transformers model (local, no API).

### Step 2: Transcript Parsing
- **LLM calls:** 8-15 (depends on chunking strategy)
- **Model:** llama-3.3-70b-versatile
- **Why 70b:** Risk extraction requires understanding nuance, anecdotes, and implied risks. The 8b model missed subtle risks and produced lower-quality descriptions. 70b correctly identifies that "we haven't cracked the sustainability angle" implies an ESG/sustainability risk even though the word "risk" isn't used.
- **Why chunked:** Groq free tier has 12K TPM (tokens per minute). The full transcript is ~4.5K tokens input + ~3K tokens output = ~7.5K tokens. This fits within 12K TPM for a single call, but Groq's rate limiter rejects it because it pre-estimates output tokens. Chunking keeps each call under ~2K total tokens.
- **Rate limit impact:** Each chunk uses ~1K input + ~500-1K output tokens. At 12K TPM, we can do ~6 calls/minute. With 4-second intervals between calls, we stay under the limit most of the time but still hit 429s on ~20% of calls.

### Step 3: Deduplication
- **LLM calls:** 1
- **Model:** llama-3.3-70b-versatile
- **Why 70b:** Dedup requires semantic reasoning — understanding that "cyber attacks on infrastructure" and "cyber is inherent" are the same underlying risk. The 8b model under-merges (treats related risks as distinct).
- **Why single call:** The dedup prompt sends a compact summary of all mentions (~50-100 lines). This fits in one call for up to ~40 mentions. Beyond that, the LLM's reasoning degrades — it creates too many fine-grained groups instead of aggressively merging.
- **Key limitation:** With 84 mentions (V2), the dedup prompt was ~4.3K tokens. The LLM produced 58 groups instead of the target 15-25. The context window isn't the issue — the LLM's ability to reason about 84 items simultaneously is.

### Step 4: Registry Mapping
- **LLM calls:** 1 per deduplicated risk (18-58 calls)
- **Model:** llama-3.3-70b-versatile
- **Why 70b:** Mapping requires understanding that a client saying "energy costs impacting our cost base" semantically matches a registry entry called "Increase in energy prices" under "Profitability & Liquidity". The 8b model assigns LOW confidence to correct matches.
- **Why per-risk:** Each risk needs its own ChromaDB query (top 3 candidates) and LLM evaluation. Batching would exceed token limits.
- **Rate limit impact:** With 4-second intervals, 18 mapping calls take ~72 seconds. 58 calls take ~4 minutes. This is the most API-intensive step.

### Steps 5-6: Review Queue & Output
- **LLM calls:** 0
- **Why:** Pure data transformation. Filtering, JSON serialization, file writing.

## Limitations

### 1. Groq Free Tier Rate Limits (Root Cause of Most Issues)
- **12K TPM** for llama-3.3-70b-versatile — forces chunking, prevents single-call parsing
- **1K RPD** — limits total daily runs to ~10-15 full pipeline executions
- **429 errors** on ~20% of calls even with 4-second intervals
- **Impact:** Chunking destroys cross-question context. The LLM can't see that "cyber" in Q1 and "cyber is inherent" in Q3 are the same risk when they're in different chunks.

### 2. Dedup Scaling Problem
- Works well for <40 mentions (V1: 68→18, good quality)
- Degrades badly for >60 mentions (V2: 84→58, poor quality)
- The LLM treats the dedup task as "light grouping" instead of "aggressive merging" when the list is long
- **Fix needed:** Two-pass dedup — first pass groups obvious duplicates, second pass merges related risks

### 3. Cascade Extraction as Standalone Risks
- Q12/Q13 describe how risks cascade into each other
- The parser extracts each cascade consequence as a separate risk mention
- V2 Q12 alone produced 15 mentions — all cascade consequences that should be metadata, not risks
- **Fix needed:** Post-extraction filter that identifies cascade-only mentions and converts them to cascade_links on existing risks

### 4. Risk Bundling
- Despite explicit "one risk per entry" instructions, the LLM occasionally bundles 2-3 risks
- V1 RISK_016: "AI governance, electronic waste, and existential business model risk"
- V2 RISK_009: "AI model errors, bias in decision-making, customer data exposure"
- **Fix needed:** Post-dedup validation that splits bundled descriptions (detect "and" joining unrelated concepts)

### 5. Similarity Scores Near Zero
- ChromaDB's default Sentence Transformers embeddings produce very different distance scales than OpenAI's text-embedding-3-large
- Most similarity scores are 0.00-0.10 even for correct matches
- The unmapped threshold (LOW confidence + similarity < 0.75) is effectively never triggered by similarity alone
- **Fix needed:** Recalibrate the threshold, or rely solely on LLM confidence for unmapped decisions

### 6. Missing cascade_links Population
- The `cascade_links` field on MappedRisk is always empty `[]`
- Cascade relationships are captured in `cascade_context` text but never resolved to actual risk IDs
- **Fix needed:** Post-mapping pass that parses cascade_context and resolves references to RISK_NNN IDs

## Recommended Improvements (Priority Order)

### P0 — Fix Dedup (Biggest Impact)
Two-pass deduplication:
1. **Pass 1:** Group obvious duplicates (same risk described in different questions)
2. **Pass 2:** Merge related sub-risks into parent risks (e.g., "GDPR enforcement" + "5G regulations" + "spectrum policy" → single "Regulatory risk" with sub-categories)

### P1 — Cascade Post-Processing
After extraction, before dedup:
- Identify mentions from Q12/Q13 that describe cascade pathways
- Instead of keeping them as standalone risks, attach them as `cascade_context` on the originating risk
- Populate `cascade_links` with resolved RISK_NNN IDs after dedup

### P2 — Upgrade LLM Provider
Switch to a provider with larger context windows and higher rate limits:
- **Gemini 2.5 Flash:** 1M token context, generous free tier — single-call parsing, no chunking needed
- **GPT-4o:** 128K context — same benefit
- **Groq paid tier:** 6K RPD instead of 1K, same model quality

### P3 — Recalibrate Similarity Threshold
- Run the registry mapper on known risk-registry pairs to establish baseline similarity scores
- Adjust the unmapped threshold based on actual ChromaDB distance distributions
- Consider using LLM confidence as the sole unmapped signal (drop similarity threshold)

### P4 — Bundle Detection & Splitting
Post-dedup validation:
- If a risk description contains "and" or commas joining 3+ distinct concepts, split into separate risks
- Use a lightweight LLM call to confirm whether the concepts are truly distinct or facets of one risk
